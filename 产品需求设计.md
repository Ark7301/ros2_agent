# 产品需求设计文档

## 面向 ROS 2 架构的 AI Agent 机器人任务调度系统

---

## 1 产品定位

一个面向 ROS 2 移动机器人的智能任务调度中间层系统。用户通过自然语言下达指令，系统自动完成任务理解、分解、规划，并通过 ROS 2 原生通信接口（Topic / Service / Action）驱动机器人执行，无需修改底层控制逻辑。

**核心价值**：
- 用自然语言替代行为树/状态机的手动配置
- AI Agent 与 ROS 2 通信机制原生适配，开箱即用
- 降低机器人任务调度的使用门槛
- 插件化架构，支持从传统导航到端到端视觉导航的平滑演进

---

## 2 用户角色

| 角色 | 描述 | 典型场景 |
|------|------|---------|
| 操作员 | 非专业开发人员，通过自然语言与机器人交互 | "去厨房拿一杯水" |
| 开发者 | ROS 2 开发人员，注册新的机器人能力和扩展系统 | 注册新的 Action Server、定义工具描述 |
| 系统管理员 | 负责系统部署、API 配置和运维 | 配置 LLM API 密钥、监控系统状态 |

## 3 系统架构总览

### 3.1 架构设计原则

1. **感知-决策-执行分离**：感知输入、智能决策（LLM）、机器人执行（ROS 2 Action）三层解耦，任意一层可独立替换
2. **Provider 插件机制**：AI 模型调用通过统一的 Provider 接口抽象，当前实现 LLMProvider，未来可插入 VLM/端到端导航 Provider
3. **ROS 2 原生适配**：通过 Topic / Service / Action 三种适配器与 ROS 2 生态无缝对接
4. **导航作为执行能力**：导航（SLAM + Nav2）是任务执行层的一种能力，而非系统核心；架构设计允许未来以端到端视觉导航（如 NaVid）替代传统 SLAM+Nav2 方案
5. **ROS 2 版本兼容性设计**：系统基于 ROS 2 Humble 开发验证，但架构设计面向跨版本兼容，确保可平滑迁移至 Jazzy 及后续发行版

### 3.2 ROS 2 版本兼容性策略

当前开发基于 ROS 2 Humble（LTS, 支持至 2027），同时为 Jazzy（LTS, 支持至 2029）及后续版本预留兼容路径：

**兼容性设计要点**：

- 仅使用 ROS 2 稳定公共 API（`rclpy` 核心接口），不依赖发行版特有的内部实现
- ROS 2 通信适配层封装所有与 ROS 2 版本相关的调用（Topic / Service / Action），上层 Agent 核心不直接调用 `rclpy`，版本差异在适配层内部吸收
- 仿真环境通过 launch 文件隔离：Humble 使用 Gazebo Classic，Jazzy 使用 Gazebo Harmonic，Agent 核心代码无需修改
- 配置文件中声明 ROS 2 版本相关参数（如 QoS 策略、DDS 配置），避免硬编码
- Nav2 / SLAM Toolbox 等依赖包通过 `package.xml` 声明版本范围，而非锁定特定版本

**Humble → Jazzy 已知差异与应对**：

| 差异项 | Humble | Jazzy | 应对策略 |
|--------|--------|-------|---------|
| 仿真器 | Gazebo Classic | Gazebo Harmonic (gz-sim) | launch 文件分版本维护，Agent 不感知仿真器差异 |
| DDS 中间件 | Fast-DDS (默认) | Fast-DDS (默认，可选 Zenoh) | 通过 RMW 环境变量切换，代码无需修改 |
| QoS 默认策略 | 部分话题默认 BEST_EFFORT | 更严格的 QoS 匹配 | 适配层显式声明 QoS Profile，不依赖默认值 |
| 消息序列化 | CDR | CDR（Iron+ 引入 Type Hash） | 使用标准消息类型，不自定义序列化 |
| Python API | rclpy (稳定) | rclpy (兼容，少量新增) | 仅使用两版本共有的稳定 API 子集 |

**目录结构体现**：

```
launch/
├── humble/                    # Humble 专用 launch 配置
│   └── gazebo_classic.launch.py
├── jazzy/                     # Jazzy 专用 launch 配置（预留）
│   └── gazebo_harmonic.launch.py
├── agent_launch.py            # Agent 核心启动（版本无关）
└── navigation_launch.py       # 导航启动（版本无关）
```

### 3.3 Provider 插件接口

系统通过抽象的 `ModelProvider` 接口实现 AI 模型的可插拔替换：

```python
class ModelProvider(ABC):
    """AI 模型提供者抽象接口"""

    @abstractmethod
    async def parse_task(self, context: TaskContext) -> TaskResult:
        """解析自然语言指令，返回结构化任务描述"""
        pass

    @abstractmethod
    async def plan_path(self, context: TaskContext) -> PlanResult:
        """路径规划，返回目标坐标或语义引导路径"""
        pass


class LLMProvider(ModelProvider):
    """V1 当前实现：基于纯文本 LLM 的任务解析
    文本指令 → OpenAI GPT Function Calling → 结构化任务 → Nav2 执行
    """
    async def parse_task(self, context: TaskContext) -> TaskResult: ...
    async def plan_path(self, context: TaskContext) -> PlanResult: ...
```

未来可通过实现新的 Provider 扩展系统能力（如 VLMProvider 支持视觉语言理解，E2ENavProvider 支持端到端视觉导航），通过配置文件切换，无需修改核心调度逻辑。

### 3.4 演进路线图

| 阶段 | 导航范式 | 地图依赖 | Provider | 参考技术 |
|------|---------|---------|----------|---------|
| V1（当前） | LLM → 坐标 → Nav2 | SLAM 几何地图 | LLMProvider | Nav2 + SLAM Toolbox |
| V2（近期） | VLM + 语义记忆 → Nav2 | SLAM 地图 + 语义层 | VLMProvider | GPT-4V / LLaVA |
| V3（远期） | 视频+指令 → 端到端动作 | 无地图（仅单目 RGB） | E2ENavProvider | NaVid (RSS 2024) / RT-2 |

V3 阶段参考 NaVid（RSS 2024）等端到端 VLN 方案：仅依赖单目 RGB 视频流 + 自然语言指令即可实现导航，无需 SLAM 地图、里程计或深度传感器。届时 SLAM 可降级为可选的安全兜底层。

## 4 功能需求

### 4.1 自然语言任务输入（P0 - 必须实现）

**FR-001 自然语言指令接收**
- 系统应提供 CLI 命令行界面接收用户自然语言输入
- 支持中文和英文指令
- 输入示例："导航到A点"、"先去A点，再去B点"、"巡逻A、B、C三个点"

**FR-002 多轮对话支持**
- 当指令不明确时，系统应主动向用户确认
- 支持上下文关联的连续指令（如"再去一次"）

### 4.2 任务解析模块（P0 - 必须实现）

**FR-003 意图识别**
- 系统应能识别以下核心意图类型：
  - `navigate_to` — 导航到指定位置
  - `patrol` — 巡逻（多点循环导航）
  - `rotate` — 原地旋转指定角度
  - `stop` — 停止当前任务
  - `status` — 查询机器人当前状态
  - `start_mapping` — 启动 SLAM 建图
  - `save_map` — 保存当前地图
  - `stop_mapping` — 停止建图
- 通过调用外部 LLM API（OpenAI GPT）实现意图识别

**FR-004 参数提取**
- 从自然语言中提取任务参数：
  - 目标位置（支持语义地名 → 坐标映射，如"厨房" → [2.0, 3.5, 0.0]）
  - 运动参数（速度、角度等）
  - 任务约束（顺序、重复次数等）

**FR-005 任务结构化输出**
- 将解析结果转化为标准化的任务描述格式（JSON）：
```json
{
  "task_id": "uuid",
  "intent": "navigate_to",
  "params": {
    "target": {"x": 2.0, "y": 3.5, "theta": 0.0},
    "target_name": "厨房"
  },
  "priority": 1,
  "timestamp": "2026-02-12T10:00:00Z"
}
```

### 4.3 任务规划模块（P0 - 必须实现）

**FR-006 单任务规划**
- 将单一意图直接映射为对应的 ROS 2 Action 调用

**FR-007 多步骤任务分解**
- 将复合指令分解为有序的子任务序列
- 示例："先去A点，然后去B点" → [navigate_to(A), navigate_to(B)]

**FR-008 任务队列管理**
- 维护待执行任务的优先级队列
- 支持任务的插入、取消和重排序

### 4.4 任务执行模块（P0 - 必须实现）

**FR-009 ROS 2 Action 客户端调度**
- 封装 ROS 2 Action Client，支持：
  - 发送目标（Goal）
  - 接收反馈（Feedback）
  - 获取结果（Result）
  - 取消任务（Cancel）
- 主要对接的 Action：
  - `NavigateToPose`（Nav2 导航）
  - `Spin`（原地旋转）
  - 可扩展注册自定义 Action

**FR-010 任务状态监控**
- 实时跟踪任务执行状态：PENDING → EXECUTING → SUCCEEDED / FAILED / CANCELLED
- 任务失败时支持自动重试（可配置重试次数）

**FR-011 执行反馈上报**
- 将 Action 执行过程中的反馈信息（如当前位置、剩余距离）转化为用户可读的文本输出

### 4.5 外部 AI 模型 API 调用模块（P0 - 必须实现）

**FR-012 LLM API 通信**
- 通过 HTTP/HTTPS 协议调用 OpenAI GPT API
- 使用 Function Calling 机制定义机器人可用能力
- 支持异步调用，不阻塞 ROS 2 节点运行

**FR-013 API 配置管理**
- 支持通过配置文件（YAML）管理 API 参数：
  - API endpoint / API key
  - 模型选择（gpt-4 / gpt-3.5-turbo 等）
  - 超时时间 / 重试策略
  - Token 限制

**FR-014 错误处理与降级**
- API 调用失败时的处理策略：
  - 自动重试（指数退避）
  - 超时降级（使用本地规则匹配作为备选）
  - 错误日志记录

### 4.6 ROS 2 通信适配层（P0 - 必须实现）

本层是实现 ROS 2 版本兼容的关键隔离层。Agent 核心通过适配层与 ROS 2 交互，不直接调用 `rclpy`，版本差异在此层内部吸收。

**FR-015 Topic 适配器**
- 订阅机器人状态话题（如 `/odom`、`/scan`、`/amcl_pose`）获取环境信息
- 发布系统状态话题供外部监控

**FR-016 Service 适配器**
- 调用 ROS 2 Service 完成即时查询操作（如获取地图信息、查询参数）

**FR-017 Action 适配器（核心）**
- 封装统一的 Action Client 接口
- 支持动态注册新的 Action 类型
- 管理 Action 的生命周期（发送目标 → 监听反馈 → 获取结果）

### 4.7 导航与地图管理模块（P0 - 必须实现）

**FR-018 SLAM 建图集成**
- 集成 SLAM Toolbox 作为默认 2D SLAM 引擎
- 支持通过自然语言触发建图操作："开始建图"、"保存地图"、"停止建图"

**FR-019 地图管理**
- 管理地图文件的加载、保存和切换（`.pgm` + `.yaml` 格式）
- 支持多地图管理（如不同楼层），通过自然语言切换："加载办公室地图"

**FR-020 定位状态监控**
- 订阅 `/amcl_pose` 获取实时定位，监控定位质量
- 定位丢失时暂停任务并尝试重定位，恢复后自动继续

**FR-021 语义地名映射**
- 维护语义地名到地图坐标的映射表（`locations.yaml`）：
```yaml
locations:
  厨房: {x: 2.0, y: 3.5, theta: 0.0}
  客厅: {x: 0.0, y: 0.0, theta: 0.0}
  卧室: {x: -1.0, y: 2.0, theta: 1.57}
  充电桩: {x: 0.0, y: 0.0, theta: 0.0}
```
- 支持通过自然语言动态标注："把这里标记为会议室" → 读取当前定位坐标并保存
- 支持运行时动态添加/修改/删除地点

### 4.8 系统监控与日志（P1 - 应该实现）

**FR-022 运行日志**
- 记录所有任务的生命周期日志（创建、执行、完成/失败）
- 记录 LLM API 调用日志（请求、响应、耗时）

**FR-023 系统状态查询**
- 用户可通过自然语言查询系统状态：
  - "机器人在哪？" → 返回当前定位坐标 + 最近语义地点
  - "当前任务是什么？" → 返回任务队列状态

### 4.9 多模态扩展（P2 - 架构预留）

**FR-024 Provider 插件扩展**
- 定义抽象 `ModelProvider` 接口，当前实现 `LLMProvider`
- 预留 `VLMProvider`（视觉语言模型）和 `E2ENavProvider`（端到端导航）接口
- 通过配置文件切换 Provider，无需修改核心调度逻辑

**FR-025 感知通道扩展**
- 定义统一的 `TaskContext` 数据结构，当前实现文本通道
- 预留图像（`/camera/image_raw`）、深度图、语音等感知通道接入点

**FR-026 空间语义记忆（预留）**
- V1 阶段：语义地名 + 坐标的静态映射（`locations.yaml`）
- 预留 V2 能力：VLM 自动构建空间语义记忆（地点 + 物体 + 场景描述），支持基于记忆的智能查询（如"冰箱在哪？"→ 查询记忆 → 导航到厨房）

### 4.10 Web 可视化界面（P2 - 可选实现）

**FR-027 Web 控制面板**
- 提供简易 Web 界面：自然语言输入、任务队列可视化、机器人位置显示、系统日志查看

## 5 非功能需求

### 5.1 性能需求

| 指标 | 要求 |
|------|------|
| 自然语言解析延迟 | ≤ 3 秒（含 API 调用） |
| 任务调度响应时间 | ≤ 500 毫秒（从解析完成到 Action 发送） |
| 系统启动时间 | ≤ 10 秒 |
| 并发任务队列容量 | ≥ 20 个待执行任务 |

### 5.2 可靠性需求

- API 调用失败自动重试，最多 3 次
- 网络断开时系统不崩溃，进入降级模式
- 任务执行失败后自动恢复到安全状态

### 5.3 可扩展性需求

- 新增机器人能力（Action）无需修改核心代码，通过配置注册
- 支持替换不同的 LLM 提供商（OpenAI / 本地模型）
- AI 模型 Provider 可插拔替换，通过配置切换
- 模块间通过 ROS 2 接口松耦合
- Agent 核心代码与 ROS 2 发行版解耦，通过适配层隔离版本差异，支持 Humble → Jazzy 及后续版本的平滑迁移

### 5.4 安全性需求

- API 密钥不硬编码，通过环境变量或加密配置文件管理
- 对用户输入进行基本的安全过滤（防止 prompt injection）

## 8 核心工作流

### 8.1 标准任务执行流程

```
用户输入 "导航到厨房"
       │
       ▼
[交互接口] 接收文本
       │
       ▼
[任务解析] 调用 LLM API (Function Calling)
       │  → 识别意图: navigate_to
       │  → 提取参数: target_name="厨房"
       │
       ▼
[语义映射] "厨房" → {x: 2.0, y: 3.5, theta: 0.0}
       │
       ▼
[任务规划] 生成任务: NavigateToPose(x=2.0, y=3.5)
       │
       ▼
[任务队列] 入队，等待调度
       │
       ▼
[任务执行] 通过 Action Adapter 发送 Goal
       │  → 监听 Feedback（当前位置、剩余距离）
       │  → 输出反馈: "正在前往厨房，剩余距离 2.3m"
       │
       ▼
[执行完成] 返回结果: "已到达厨房"
```

### 8.2 多步骤任务流程

```
用户输入 "先去厨房，再去客厅，最后回到充电桩"
       │
       ▼
[任务解析] LLM 分解为 3 个子任务
       │
       ▼
[任务规划] 生成有序任务序列:
       │  1. navigate_to(厨房)
       │  2. navigate_to(客厅)
       │  3. navigate_to(充电桩)
       │
       ▼
[任务队列] 按序入队
       │
       ▼
[任务执行] 依次执行，每个完成后自动触发下一个
```

## 9 验收标准

### 9.1 功能验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-01 | 单点导航 | 输入"去厨房"，机器人成功导航到厨房坐标 |
| AC-02 | 多点导航 | 输入"先去A再去B"，机器人按序完成导航 |
| AC-03 | 巡逻任务 | 输入"巡逻A、B、C"，机器人循环巡逻 |
| AC-04 | 任务取消 | 执行中输入"停止"，机器人停止当前任务 |
| AC-05 | 状态查询 | 输入"机器人在哪"，返回当前位置 + 最近语义地点 |
| AC-06 | 错误恢复 | 导航失败后系统自动重试或报告错误 |
| AC-07 | API 降级 | 断开网络后系统不崩溃，提示 API 不可用 |
| AC-08 | SLAM 建图 | 输入"开始建图"，SLAM 启动并生成栅格地图 |
| AC-09 | 地图保存加载 | 输入"保存地图"，地图持久化；重启后可加载 |
| AC-10 | 动态地点标注 | 输入"把这里标记为会议室"，当前坐标保存为语义地点 |
| AC-11 | 定位丢失恢复 | 定位丢失时系统暂停任务并尝试重定位 |

### 9.2 性能验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-12 | 解析延迟 | 自然语言到任务结构化 ≤ 3 秒 |
| AC-13 | 调度延迟 | 任务结构化到 Action 发送 ≤ 500ms |
| AC-14 | 连续任务 | 连续执行 10 个导航任务无异常 |
| AC-15 | 定位精度 | AMCL 定位误差 ≤ 0.3m（仿真环境） |
