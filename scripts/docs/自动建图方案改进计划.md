# 自动建图方案改进计划

## 现状问题

当前 `auto_explore.py` 是自研的简易 Wavefront Frontier 探索，存在：
- 卡住后恢复慢（超时 15s + Nav2 recovery 耗时）
- frontier 选点质量差（选到不可达点、墙角、costmap 膨胀区）
- 无 frontier 黑名单机制（失败的目标被重复选中）
- 无 costmap 可达性验证（直接用 OccupancyGrid 选点，不考虑 costmap 代价）

## 成熟方案调研

### 1. m-explore-ros2 (explore_lite ROS2 移植)
- 来源：[robo-friends/m-explore-ros2](https://github.com/robo-friends/m-explore-ros2)
- 核心：基于 costmap_2d 的 frontier 检测，而非原始 OccupancyGrid
- 关键机制：
  - 用 Nav2 costmap 做 frontier 检测（自动排除膨胀区内的点）
  - frontier 聚类（相邻 frontier cell 合并为一个 cluster，选 cluster 中心）
  - 通过 `makePlan` 验证可达性（不可达的 frontier 直接跳过）
  - 内置 frontier 黑名单（导航失败的 frontier 加入黑名单不再选择）
- 缺点：C++ 实现，需要编译安装，且对 ROS2 Jazzy 兼容性未验证

### 2. AniArka/Autonomous-Explorer-and-Mapper
- 来源：[GitHub](https://github.com/AniArka/Autonomous-Explorer-and-Mapper-ros2-nav2)
- 核心：Python 实现，结构简单，与我们的方案类似
- 特点：kidnapped robot recovery（定位丢失恢复）
- 缺点：无黑名单、无 costmap 验证、无聚类，和我们的问题一样

### 3. explore_lite 原版核心算法（参考价值最高）
- frontier 检测在 costmap 上做（不是 OccupancyGrid）
- frontier 聚类：BFS 找连通的 frontier cell，合并为 cluster
- 目标选择：cluster size 加权 + 距离加权（大 cluster 优先）
- 可达性验证：调用 Nav2 `ComputePathToPose` 检查是否能规划路径
- 黑名单：`makePlan` 失败或导航失败的 frontier 加入黑名单，定期清理

## 改进计划

### 方案：借鉴 explore_lite 核心机制，重写 auto_explore.py

不引入外部依赖（不编译 m-explore-ros2），而是在现有 Python 代码中实现关键机制：

#### P0：frontier 黑名单 + 快速脱困
- 导航失败/超时的目标坐标加入黑名单（半径 1m 内不再选择）
- 黑名单有 TTL（5 分钟后过期，允许重试）
- 超时后调用 Nav2 `Backup` behavior 后退 0.3m 脱困

#### P1：costmap 可达性验证
- 选定 frontier 后，调用 Nav2 `ComputePathToPose` action 验证
- 规划失败的 frontier 直接加入黑名单，选下一个
- 避免发送不可达的导航目标

#### P2：frontier 聚类
- BFS 找连通的 frontier cell，合并为 cluster
- 选 cluster 中心点（而非单个 frontier cell）
- 按 cluster size × 距离倒数 排序，优先选大且近的 cluster

#### P3：costmap 上做 frontier 检测
- 订阅 `/global_costmap/costmap`（而非 `/map`）
- costmap 已包含膨胀信息，frontier 天然远离障碍物
- 消除手动 WALL_CHECK_R 的需要

### 实施顺序
1. P0（黑名单 + 脱困）— 解决当前卡住问题，立即可做
2. P1（可达性验证）— 减少无效导航，需要调用 Nav2 service
3. P2（聚类）— 提升选点质量
4. P3（costmap frontier）— 最终方案，需要重构 frontier 检测逻辑

### 预估工作量
- P0：1 小时
- P1：2 小时
- P2：1 小时
- P3：3 小时
