# 自动场景建图开源方案调研

调研目标：为 MOSAIC 的 SLAM → 场景图自动构建链路选择最合适的开源方案。

## 方案总览

| 方案 | 层级 | 输入 | 输出 | 实时性 | 与 MOSAIC 契合度 |
|------|------|------|------|--------|-----------------|
| slam_toolbox + Nav2 | 2D 占据栅格 | LiDAR | .pgm + .yaml 地图 | 实时 | ⭐⭐⭐ 直接对接 MapAnalyzer |
| NVIDIA cuVSLAM + nvblox | 3D 体素 + 2D costmap | 立体相机 | TSDF 体素 + Nav2 costmap | 实时 | ⭐⭐⭐ NVIDIA 官方方案 |
| Kimera / Hydra (MIT SPARK) | 3D 动态场景图 | VI-SLAM | 分层 DSG (Room→Object) | 实时 | ⭐⭐⭐⭐ 层级结构完美匹配 |
| ConceptGraphs | 开放词汇 3D 场景图 | RGB-D + 位姿 | 节点=物体 + 语义特征 | 离线 | ⭐⭐⭐ VLM 语义填充参考 |
| HOV-SG | 层次化开放词汇场景图 | RGB-D + 里程计 | Floor→Room→Object 层级 | 离线 | ⭐⭐⭐⭐ 层级 + 开放词汇 |
| DovSG | 动态开放词汇场景图 | RGB-D | 可增量更新的场景图 | 近实时 | ⭐⭐⭐⭐ 动态更新 + 操作 |
| OK-Robot | 零样本移动操作 | RGB-D 扫描 | 语义点云地图 | 离线建图 | ⭐⭐ 系统集成参考 |

---

## 1. slam_toolbox + Nav2（基础层，推荐 Phase 1）

来源：[slam_toolbox](https://github.com/SteveMacenski/slam_toolbox) + [Nav2](https://docs.nav2.org/)

这是 ROS2 生态中最成熟的 2D SLAM 方案，MOSAIC 的 MapAnalyzer 已经为此设计。

流程：LiDAR /scan → slam_toolbox 在线建图 → map_saver_cli 保存 .yaml+.pgm → MapAnalyzer 加载 → 连通域分析提取房间拓扑 → SceneGraphManager.merge_room_topology()

优势：
- Nav2 官方推荐，文档完善，Isaac Sim 有现成教程
- 输出标准 OccupancyGrid 格式，MapAnalyzer 已实现解析
- 支持在线建图（mapping）和定位（localization）双模式
- Nova Carter 在 Isaac Sim 中有 LiDAR 传感器，直接可用

局限：
- 仅 2D 占据栅格，无语义信息
- 房间分割依赖 MapAnalyzer 的连通域分析（质量取决于地图质量）
- 无法识别物体和家具

MOSAIC 集成点：MapAnalyzer.load_map() → extract_room_topology() → SceneGraphManager.merge_room_topology()

## 2. NVIDIA cuVSLAM + nvblox（推荐 Phase 1 备选）

来源：[Isaac ROS Visual SLAM](https://nvidia-isaac-ros.github.io/concepts/visual_slam/cuvslam/tutorial_isaac_sim.html) + [nvblox](https://nvidia-isaac-ros.github.io/repositories_and_packages/isaac_ros_nvblox/index.html)

NVIDIA 官方的视觉 SLAM + 3D 重建方案，是 Isaac Perceptor 的核心组件。

流程：立体相机 → cuVSLAM 视觉定位 → nvblox 3D 体素重建 → 输出 2D costmap 给 Nav2 + 3D mesh 可视化

优势：
- GPU 加速，实时性能优秀
- 无需 LiDAR，纯视觉方案（Nova Carter 多目摄像头直接可用）
- 与 Isaac Sim ROS2 Bridge 深度集成
- 输出 2D costmap 可直接给 Nav2 做路径规划
- 支持保存/加载地图（~/save_map, ~/load_map 服务）

局限：
- 输出是 TSDF 体素而非标准 .pgm 占据栅格，需要转换才能给 MapAnalyzer
- 依赖 NVIDIA GPU（Jetson 或 discrete GPU）
- 文档主要面向 Jetson 平台，桌面 GPU 需要额外配置

MOSAIC 集成点：需要新增 nvblox → OccupancyGrid 转换层，或直接使用 nvblox 的 2D costmap 输出

## 3. Kimera / Hydra（MIT SPARK Lab，推荐 Phase 2）

来源：[Kimera](https://github.com/MIT-SPARK/Kimera) + [Hydra](https://github.com/MIT-SPARK/Hydra) + [Spark-DSG](https://github.com/MIT-SPARK/Spark-DSG)

MIT SPARK Lab 的 3D 动态场景图系统，是学术界场景图 SLAM 的标杆。

Kimera 是第一个从视觉惯性数据自动构建 3D 动态场景图（DSG）的系统。Hydra 是其实时版本，能增量构建分层场景图。

层级结构（与 MOSAIC 高度契合）：
- Layer 1: 3D Mesh（度量层）
- Layer 2: Objects（物体节点）
- Layer 3: Places（可通行区域）
- Layer 4: Rooms（房间，通过 Places 聚类）
- Layer 5: Buildings（建筑）

流程：RGB-D + IMU → Kimera-VIO（视觉惯性里程计）→ Kimera-Semantics（语义 3D 重建）→ Hydra（实时场景图构建）→ 分层 DSG

优势：
- 层级结构（Building→Room→Object）与 MOSAIC 的 SceneGraph 三层结构完美对应
- 实时增量构建，支持动态环境
- 开源 C++ 实现，有 Python 绑定（Spark-DSG）
- 自动房间分割（基于 Places 层的聚类），比 MapAnalyzer 的连通域分析更精确
- 学术引用量高，持续维护

局限：
- C++ 为主，ROS1 支持更成熟，ROS2 移植进行中
- 依赖视觉惯性传感器（VIO），Isaac Sim 需要配置 IMU
- 语义分割依赖预训练模型（如 Mask2Former），非开放词汇
- 部署复杂度较高

MOSAIC 集成点：Spark-DSG Python API → 直接映射到 SceneGraph 的 Room/Furniture/Object 节点

## 4. ConceptGraphs（ICRA 2024，推荐 VLM 语义填充参考）

来源：[ConceptGraphs](https://concept-graphs.github.io/) + [GitHub](https://github.com/concept-graphs/concept-graphs)

MIT + Université de Montréal 的开放词汇 3D 场景图方案。

核心思路：利用 2D 基础模型（SAM 分割 + CLIP 特征）的输出，通过多视角关联融合到 3D，构建开放词汇场景图。然后用 LLM 生成物体描述和物体间关系（边）。

流程：RGB-D 序列 + 位姿 → SAM 实例分割 → CLIP 特征提取 → 多视角 3D 融合 → LLM 生成描述和关系 → 开放词汇 3D 场景图

优势：
- 开放词汇：不限于预定义类别，能识别任意物体
- 利用 2D 基础模型（SAM + CLIP），无需 3D 训练数据
- 场景图结构（节点=物体，边=空间关系）与 MOSAIC 的 SceneGraph 直接对应
- 支持语言查询（"找到红色的杯子"）

局限：
- 离线处理，非实时（每帧需要 SAM + CLIP 推理）
- 无房间层级（只有物体层），需要额外的房间分割
- 依赖准确的相机位姿（需要外部 SLAM 提供）
- GPU 内存需求大（SAM + CLIP + LLM）

MOSAIC 集成点：ConceptGraphs 的物体检测 + 关系推理流程可作为 SceneAnalyzer + SceneGraphBuilder 的参考实现

## 5. HOV-SG（RSS 2024，推荐 Phase 3）

来源：[HOV-SG](https://hovsg.github.io/) + [GitHub](https://github.com/hovsg/HOV-SG)

Freiburg 大学的层次化开放词汇 3D 场景图方案，专为语言引导的机器人导航设计。

核心创新：自上而下构建层次化场景图（Floor → Room → Object），每层都有开放词汇特征。支持多层建筑和跨楼层导航。

流程：RGB-D + 里程计 → 开放词汇 3D 分割 → 楼层分割 → 房间分割（Voronoi + 语义聚类）→ 物体检测 → 层次化场景图

优势：
- 层级结构（Floor→Room→Object）与 MOSAIC 完美匹配
- 开放词汇：房间名称和物体类别都是开放的
- 比 ConceptGraphs 多了房间层级，比 Kimera 多了开放词汇
- 表示大小比密集开放词汇地图减少 75%
- 在真实多层环境中验证了长时程语言导航

局限：
- 离线处理（非实时）
- 依赖 OpenSeg/LSeg 等开放词汇分割模型
- 房间分割算法较复杂（Voronoi + 语义聚类）
- 代码成熟度不如 Kimera

MOSAIC 集成点：HOV-SG 的房间分割算法可替代 MapAnalyzer 的连通域分析；开放词汇特征可增强 SpatialProvider 的语义匹配

## 6. DovSG（动态开放词汇场景图）

来源：[DovSG](https://bjhyzj.github.io/dovsg-web) + [arXiv 2410.11989](https://arxiv.org/html/2410.11989)

专为长时程语言引导移动操作设计的动态场景图方案。

核心创新：支持增量更新——机器人与环境交互后（如拿起物品），场景图能局部更新而无需全局重建。

流程：RGB-D → VLM 物体检测 → 3D 场景图构建 → 匈牙利匹配跟踪 → 局部增量更新

优势：
- 动态更新：与 MOSAIC 的 update_from_execution() 理念一致
- 开放词汇物体检测
- 支持长时程操作任务
- 局部更新效率高

局限：
- 论文较新（2024.10），代码成熟度待验证
- 无房间层级

MOSAIC 集成点：DovSG 的增量更新机制可参考优化 SceneGraphBuilder.merge_detections() 的去重和更新逻辑

---

## 7. SayPlan（LLM + 3D 场景图任务规划）

来源：[SayPlan](https://arxiv.org/abs/2307.06135)

不是建图方案，而是 LLM 如何利用 3D 场景图进行任务规划的参考。

核心思路：利用场景图的层次结构减少 LLM 的 token 消耗——先在压缩的全局图上规划，再在任务相关子图上细化。

与 MOSAIC 的关系：
- SayPlan 的"语义子图搜索"与 SceneGraphManager.get_task_subgraph() 思路一致
- SayPlan 的"迭代重规划"与 PlanVerifier 的验证反馈机制互补
- 验证了 3D 场景图 + LLM 规划的可行性

## 推荐的分阶段集成路线

### Phase 1（当前）：slam_toolbox 2D 建图
- 使用 slam_toolbox 在 Isaac Sim 中建图
- MapAnalyzer 解析 .pgm 提取房间拓扑
- 这是 slam-simulation-pipeline spec 正在做的事

### Phase 2：Kimera/Hydra 3D 场景图
- 替换 slam_toolbox 为 Hydra，获得自动房间分割
- Spark-DSG Python API 直接映射到 MOSAIC SceneGraph
- 需要 Isaac Sim 配置 IMU + 立体相机

### Phase 3：HOV-SG 开放词汇层级
- 在 Hydra 基础上加入开放词汇特征
- 房间名称和物体类别自动识别（替代 home.yaml 手工配置）
- 与 MOSAIC 的 VLM 语义标注流水线（SceneAnalyzer）融合

### Phase 4：ConceptGraphs + DovSG 动态语义
- ConceptGraphs 的物体关系推理增强 SceneGraph 的边
- DovSG 的增量更新机制优化 SceneGraphBuilder
- 实现完全自动的场景图构建和维护

---

## 对 slam-simulation-pipeline spec 的建议

当前 spec 的设计（slam_toolbox + MapAnalyzer）是正确的 Phase 1 选择。建议补充：

1. MapAnalyzer 输出格式预留扩展点，后续可接入 Hydra 的 DSG 输出
2. SceneGraphBuilder 的 merge_detections() 参考 DovSG 的增量更新策略
3. 配置中预留 `slam.backend` 字段（slam_toolbox / cuvslam / hydra），方便后续切换
4. Isaac Sim 启动脚本中同时配置 LiDAR 和立体相机话题，为 Phase 2 做准备
