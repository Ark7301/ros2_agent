# 场景图集成 — 用户仿真体验测试指南

模拟真实用户通过自然语言与 MOSAIC 交互，验证场景图集成后的端到端体验效果。

## 启动方式

```bash
# v2 Gateway（场景图已集成）
export MINIMAX_API_KEY="your-key"
python3 -c "from mosaic.gateway.server import main; main()"

# v1 Demo（对照组，无场景图）
python3 -m mosaic_demo.main
```

启动后进入 `>>> ` 交互界面，输入自然语言指令即可。

---

## 场景 1：单步导航 — 场景图感知

验证 LLM 能基于场景图理解空间布局。

| 用户输入 | 预期行为 |
|---------|---------|
| `去厨房` | 调用 navigate_to(厨房)，场景图更新 agent AT 厨房 |
| `去卧室` | 调用 navigate_to(卧室)，AT_Edge 切换到卧室 |
| `去阳台` | 调用 navigate_to(阳台) |

观察点：
- 日志中应出现 `🔧 调用工具: navigate_to(...)` 
- 无 `⚠️ 计划验证失败` 提示（因为房间间有 REACHABLE 连接）

---

## 场景 2：复合指令 — 计划验证生效

验证 PlanVerifier 在执行前检查物理可行性。

| 用户输入 | 预期行为 |
|---------|---------|
| `帮我去厨房拿杯咖啡送过来` | LLM 分解为多步：navigate_to → operate_appliance → wait_appliance → pick_up → navigate_to → hand_over |
| `帮我去卫生间拿毛巾` | navigate_to(卫生间) → pick_up(毛巾) → navigate_to(用户位置) → hand_over(毛巾) |

观察点：
- 如果 LLM 跳步（如直接 pick_up 不在身边的物品），应看到 `⚠️ 计划验证失败` 日志
- 验证失败后 LLM 会收到反馈并重新规划，最终生成正确顺序的计划

---

## 场景 3：物品操作 — 场景图状态更新

验证执行后场景图正确反映世界状态变化。

| 用户输入 | 预期行为 |
|---------|---------|
| `去客厅拿茶几上的水杯` | 执行后 agent 持有水杯（HOLDING 边），水杯从茶几移除（ON_TOP 边删除） |
| `把水杯放下` | hand_over 后 HOLDING 边移除 |

观察点：
- 第一次拿水杯后，如果再次要求"拿水杯"，LLM 应知道已经拿着了（场景图已更新）

---

## 场景 4：空间推理 — 场景图增强 LLM 理解

验证 LLM 利用场景图进行空间推理。

| 用户输入 | 预期行为 |
|---------|---------|
| `咖啡机在哪里？` | LLM 应回答"厨房的料理台上"（来自场景图层次信息） |
| `从客厅怎么去卫生间？` | LLM 应回答经过厨房（客厅→厨房→卫生间，基于 REACHABLE 连接） |
| `茶几上有什么？` | LLM 应列出水杯、遥控器（来自场景图 CONTAINS 关系） |

观察点：
- 这些问题不需要工具调用，LLM 直接从 system prompt 中的场景图文本回答
- 回答应包含具体的空间关系信息，而非泛泛而谈

---

## 场景 5：降级兼容 — 无场景图模式

验证场景图组件失败时系统仍可正常运行。

测试方法：临时修改 `config/mosaic.yaml` 中 `scene_graph.environment_config` 为不存在的路径，重启。

| 用户输入 | 预期行为 |
|---------|---------|
| `导航到厨房` | 仍然调用 navigate_to，但无计划验证和场景图注入 |
| `你好` | 正常文本回复，系统不崩溃 |

观察点：
- 启动日志应有 `场景图初始化失败，降级为无场景图模式`
- 功能退化但不中断

---

## 场景 6：多轮对话 — 上下文连贯性

验证场景图状态在多轮对话中保持一致。

按顺序输入：
1. `去厨房` → agent 移动到厨房
2. `启动咖啡机` → operate_appliance(咖啡机, 启动)
3. `等咖啡做好` → wait_appliance(咖啡机)
4. `拿起咖啡` → pick_up(咖啡)
5. `送到客厅给用户` → navigate_to(客厅) → hand_over(咖啡)

观察点：
- 每步执行后场景图状态应正确更新
- 第 4 步不应触发验证失败（因为 agent 已在厨房）
- 第 5 步 LLM 应知道需要先导航再递交

---

## 场景 7：错误指令 — 验证反馈机制

验证不可行指令的处理。

| 用户输入 | 预期行为 |
|---------|---------|
| `拿起冰箱` | 冰箱不是 graspable 物品，验证应失败或 LLM 拒绝 |
| `去火星` | 场景图中无此位置，SpatialProvider 抛异常，LLM 应告知无法到达 |

## 对比测试：v1 vs v2

同一指令分别在 v1 Demo 和 v2 Gateway 中执行，对比差异：

| 指令 | v1 Demo（无场景图） | v2 Gateway（有场景图） |
|------|-------------------|---------------------|
| `帮我做杯咖啡送过来` | LLM 直接生成步骤，可能跳步 | PlanVerifier 验证后执行，不可行时自动修正 |
| `茶几上有什么？` | LLM 凭训练知识猜测 | LLM 基于场景图精确回答 |
| `去厨房拿水杯` | 可能直接 pick_up（物理不可行） | 先 navigate_to 再 pick_up（验证通过） |

## 日志关键标记

交互过程中关注以下日志输出：

| 标记 | 含义 |
|------|------|
| `🔄 Turn 开始` | 新一轮 ReAct 循环启动 |
| `🤖 LLM 推理中...` | 正在调用 LLM |
| `🔧 调用工具: xxx(...)` | LLM 决定执行物理动作 |
| `⚠️ 计划验证失败` | PlanVerifier 检测到不可行计划，LLM 将重新规划 |
| `✓ xxx: 成功消息` | 工具执行成功 |
| `✗ xxx: 错误消息` | 工具执行失败 |
| `✅ Turn 完成` | 本轮交互结束 |
