## 二、与本课题有关的国内外研究情况

### 2.1 ROS 2 任务调度技术研究现状

ROS 2（Robot Operating System 2）自 2017 年正式发布以来，已成为机器人系统开发的标准平台。Macenski 等人[3]全面介绍了 ROS 2 的设计理念与系统架构，指出 ROS 2 基于 DDS 通信中间件，提供了 Topic（话题）、Service（服务）和 Action（动作）三种核心通信机制，为机器人系统的模块化开发提供了灵活的接口基础。

在任务调度方面，当前 ROS 2 生态主要依赖两种传统范式：

**行为树（Behavior Tree）**：Colledanchise 和 Ögren[18]系统介绍了行为树在机器人和 AI 中的应用。Macenski 等人[6]在 Navigation2 框架中采用 BehaviorTree.CPP 作为任务调度引擎，通过 XML 配置文件定义导航行为的执行逻辑。然而，行为树存在配置复杂度高、与具体任务场景紧耦合、非专业人员无法使用等问题。一个包含导航、巡逻、异常处理的复合任务可能需要数十个节点和上百条转移规则。

**有限状态机（FSM）**：FlexBE 等基于分层状态机的行为引擎提供了可视化编辑工具，但 FSM 在处理复杂任务时存在状态爆炸问题，随着任务复杂度增加，状态数量和转移条件呈指数级增长。

总体而言，当前 ROS 2 生态中的任务调度技术仍以行为树和有限状态机为主，缺乏基于自然语言理解的智能调度方案。

### 2.2 AI Agent 技术与 LLM 工具调用范式演进

AI Agent 是指具备自主感知、推理、规划和执行能力的智能体系统。近年来，随着大语言模型（LLM）的突破性进展，基于 LLM 的 AI Agent 成为研究热点。

**LLM 工具调用范式经历了四个阶段的演进**：（1）文本解析阶段（2022 年以前），通过 Prompt Engineering 让 LLM 输出特定格式文本，依赖脆弱的正则匹配；（2）Function Calling 阶段（2023），OpenAI 在 GPT 系列中引入结构化函数调用机制[14]，首次实现 LLM 与外部工具的可靠交互；（3）ReAct Agent Loop 阶段（2023-2024），Yao 等人[13]提出将推理与行动交替进行的 Agent 范式，使 LLM 能在循环中持续推理、调用工具、观察结果、调整策略；（4）平台级 Agent 基础设施阶段（2025），OpenAI 发布 Agents SDK，Anthropic 发布 MCP 协议，Google 发布 A2A 协议，Agent Loop 从框架层面上升到平台基础设施层面。

这四个阶段并非替代关系，而是层次递进的包含关系：Function Calling 是 Agent Loop 的执行基元，Agent Loop 是平台级 Agent 的核心循环。

### 2.3 面向机器人的 LLM Agent 研究

在机器人领域，将 AI Agent 技术应用于任务规划已成为研究热点，形成了开环规划与闭环规划两条技术路线：

**开环规划**：Ahn 等人[11]提出的 SayCan 方法将 LLM 的语言理解能力与机器人的物理能力（affordance）相结合，通过可行性评分机制确保 LLM 生成的任务计划在物理世界中可执行。Liang 等人[12]提出的 Code as Policies 方法利用 LLM 直接生成机器人控制代码。然而，这些方法在执行过程中不根据环境反馈调整计划。

**闭环规划**：Huang 等人提出的 Inner Monologue（2022）在 SayCan 基础上引入多源反馈机制（成功检测、场景描述、人类交互），LLM 根据反馈动态调整后续计划，实现了机器人领域的 Agent Loop。Dalal 等人提出的 BrainBody-LLM（2024）借鉴人类神经系统的分层结构，将规划分为高层 LLM 任务分解和低层 LLM 动作生成两层，通过仿真器反馈实现闭环纠错。

**环境表征与规划 Grounding**：Rana 等人提出的 SayPlan（CoRL 2023）首次将 3D 场景图引入 LLM 规划，通过语义搜索机制在层级化场景图上定位任务相关区域。Booker 等人提出的 EmbodiedRAG（2024）将 RAG 范式适配到具身智能领域，通过任务驱动的 3D 场景图子图检索自动提取相关上下文，实验表明可将 token 消耗降低 90%，规划时间减少 70%。

**反幻觉研究**：Wan 等人提出的 EmbodiedAgent（2025）聚焦于 LLM 在机器人规划中的幻觉问题，通过监督微调使 LLM 学习在遇到不可行场景时主动报告错误。Ekpo 等人提出的 VeriGraph（2024）通过场景图约束检查验证 LLM 生成计划的可执行性。

然而，上述研究大多针对特定机器人平台设计，尚未形成与 ROS 2 通信机制深度集成的通用方案。

### 2.4 VLA 技术与具身智能前沿

VLA（Vision-Language-Action）模型是将视觉感知、自然语言理解和动作生成统一在单一学习框架中的多模态模型，代表了具身智能的前沿方向。

Google DeepMind 的 RT-2（2023）首次证明可将网络规模的视觉-语言知识迁移到机器人控制。Stanford 的 OpenVLA（2024）是首个高质量开源 VLA 模型，以 7B 参数超越 RT-2-X（55B）。Physical Intelligence 的 π0 系列（2024-2025）采用 Flow Matching 动作生成架构，实现了跨形态通用策略。Figure AI 的 Helix（2025）和 NVIDIA 的 GR00T N1（2025）采用受人类认知启发的双系统架构（慢思考 VLM + 快思考运动策略），解决了大模型推理延迟与实时控制需求的矛盾。

2025-2026 年的重要趋势是世界模型（World Model）与 VLA 的融合。WorldVLA（阿里达摩院，2025）将 VLA 和世界模型统一到单一自回归框架，两个分支互相增强。NVIDIA Cosmos 系列将世界模型定位为通用物理 AI 基础设施，为 VLA 训练提供合成数据和仿真评估环境。

VLA 技术的快速发展为本课题的系统架构设计提供了重要的演进方向——系统需要预留从传统导航方案向端到端 VLA 方案平滑过渡的能力。

### 2.5 全局状态管理与记忆架构研究

LLM 驱动的机器人规划系统面临环境表征丰富性与 LLM 上下文窗口有限性之间的核心矛盾。Liu 等人（2024）指出 LLM 对输入中间位置信息的关注度显著低于首尾位置（"Lost in the Middle"现象），Shi 等人（ICML 2023）证实 LLM 易被无关上下文分散注意力。

针对这一问题，学界提出了多种方案：EmbodiedRAG 的任务驱动子图检索、ReMEmbR（NVIDIA, 2024）的长时时空语义记忆、GraphRAG（Microsoft, 2024）的知识图谱增强检索、Clio（MIT, 2024）的实时任务驱动场景图压缩。这些方案从不同角度解决了大规模环境下 LLM 规划的上下文管理问题，为本课题的全局状态管理模块设计提供了重要参考。

### 2.6 国内外研究现状小结

综合以上分析，当前研究存在以下不足：（1）现有 AI Agent 框架多面向通用计算任务设计，尚未针对 ROS 2 的原生通信模型进行专门适配；（2）面向机器人的 Agent 研究虽然验证了技术可行性，但缺乏与 ROS 2 系统的深度集成方案；（3）ROS 2 社区目前缺少一个开箱即用的、基于 LLM 的通用任务调度中间层。本课题将在上述研究基础上，设计面向 ROS 2 原生通信模型的 AI Agent 任务调度系统。

## 三、本课题研究的主要内容及方法

### 3.1 研究内容

本课题设计并实现一个名为 MOSAIC（Modular Orchestration System for Agent-driven Intelligent Control）的面向 ROS 2 架构的 AI Agent 机器人任务调度系统。主要研究内容包括：

**（1）模块化 Agent 调度架构设计**

设计面向 ROS 2 的三层架构：用户交互层 → AI Agent 核心层 → 能力模块层 + ROS 2 基础设施。核心设计原则包括：依赖倒置（Agent 核心仅依赖抽象接口）、感知-决策-执行分离、能力插件化（Capability Plugin）。Agent 核心层包含任务解析（TaskParser）、任务规划（TaskPlanner）、任务执行（TaskExecutor）三个核心模块，通过 CapabilityRegistry 实现能力的动态注册与发现。

**（2）基于 LLM Function Calling 的自然语言任务解析**

基于 OpenAI GPT 系列 API 的 Function Calling 机制，将 ROS 2 系统中可用的机器人能力注册为 LLM 可调用的工具函数。Function Calling 的"函数名 + 参数"模式与 ROS 2 Action 的"Goal 类型 + 参数"模式天然匹配。系统从 CapabilityRegistry 动态生成 Function Calling schema，新增能力无需修改解析逻辑。

**（3）LLM 驱动的任务规划与闭环重规划**

TaskPlanner 作为系统的决策中枢，基于 LLM 常识推理生成执行计划，通过 SceneGraphValidator 验证计划可执行性。系统设计了执行中反馈驱动的决策流：Capability 上报 FeedbackEvent → 全局状态更新 → TaskPlanner 评估并触发 LLM 重规划，实现从开环规划到闭环规划的演进。

**（4）ARIA 多层记忆与 EmbodiedRAG 检索架构**

设计 ARIA（Agent with Retrieval-augmented Intelligence Architecture）三层记忆架构：工作记忆（机器人实时状态）、语义记忆（3D 场景图 + 向量嵌入索引）、情景记忆（任务执行历史经验）。参考 EmbodiedRAG 方案，实现任务驱动的场景图子图检索，使 LLM 规划时获得精简、相关的环境上下文，解决大规模环境下的 token 限制和注意力偏差问题。

**（5）3D 场景图构建与环境表征**

参考 SayPlan 的 3D Scene Graph 方案，设计"SLAM 空间骨架 + VLM 语义填充"的融合构建流程：SLAM Toolbox 构建栅格地图并提取空间拓扑 → 连通区域分析自动分割房间 → VLM 分析 RGB 图像识别物体和空间关系 → LLM 归纳语义属性 → 组装层级场景图（Building → Floor → Room → Object）。

**（6）ROS 2 通信适配层与能力插件实现**

设计 ROS 2 通信适配层，封装 Topic / Service / Action 三种通信机制，隔离 ROS 2 版本差异。实现导航模块（NavigationCapability，封装 Nav2）、建图模块（MappingCapability，封装 SLAM Toolbox）、运动模块（MotionCapability）等能力插件，以及语义地名服务（LocationService）。

**（7）系统集成与仿真验证**

在 Gazebo 仿真环境中搭建 TurtleBot3 测试平台，验证系统的端到端功能：自然语言指令 → 任务解析 → 规划 → 执行 → 结果反馈。

### 3.2 研究方法

**文献研究法**：系统调研 ROS 2 任务调度、AI Agent、LLM 工具调用、具身智能等领域的国内外研究成果，明确技术路线。

**系统设计法**：采用面向对象设计方法和依赖倒置原则，设计模块化、可扩展的系统架构。通过抽象接口定义模块间契约，确保各模块可独立替换。

**原型验证法**：采用分阶段实现策略——先构建 Mock 能力的 Demo 骨架验证 Agent 调度管道，再逐步接入真实 ROS 2 通信和仿真环境。

**属性测试法**：使用 Hypothesis 库进行属性测试，对核心正确性属性（Registry round-trip、状态机合法转换、序列化 round-trip 等）进行形式化验证。

## 四、本课题所需要解决的问题

### 4.1 AI Agent 与 ROS 2 通信机制的适配问题

ROS 2 的核心通信机制（Topic 发布/订阅、Service 请求/响应、Action 目标/反馈/结果）具有独特的异步、有状态特性，与通用 Agent 框架中的"调用函数-获取返回值"模式存在本质差异。特别是 ROS 2 Action 接口具有长时运行、可取消、持续反馈等特性（如 NavigateToPose 可能执行数十秒并持续上报位置反馈），通用 Agent 框架缺乏对这类长时异步任务的生命周期管理能力。需要设计专用的通信适配层，将 ROS 2 的三种通信模式封装为 Agent 可调用的统一接口。

### 4.2 LLM 规划的可靠性与反幻觉问题

LLM 在生成机器人任务计划时可能产生幻觉——当任务在物理上不可行时（如目标位置不存在、机器人缺乏所需能力），LLM 仍可能生成看似合理但实际不可执行的规划序列。需要设计 SceneGraphValidator 等验证机制，基于实际环境状态对 LLM 生成的计划进行形式化验证，并在验证不通过时携带反馈让 LLM 重新生成。

### 4.3 大规模环境下的上下文管理问题

随着机器人操作环境从单房间扩展到多楼层建筑，3D 场景图中的实体数量可能从数十增长到数千。直接将完整场景图输入 LLM 会导致 token 超限、注意力偏差和规划延迟增长。需要设计基于 EmbodiedRAG 的任务驱动检索机制，使 LLM 规划时仅获取精简、相关的环境子图。

### 4.4 系统可扩展性与能力热插拔问题

系统需要支持机器人能力的动态扩展——新增一个硬件模块（如机械臂、摄像头）后，应能以最低成本集成到调度系统中。需要设计基于依赖倒置的插件化架构，使新增能力只需实现 Capability 接口并注册到 Registry，Agent 核心代码零修改。同时需要预留从传统导航方案（Nav2）向端到端 VLA 方案平滑过渡的架构扩展点。

### 4.5 外部 LLM API 的实时性与可靠性问题

外部 API 调用涉及网络通信延迟（通常 1-5 秒），且存在调用失败的风险。需要设计异步通信机制（基于 Python asyncio）避免阻塞 ROS 2 节点运行，并实现指数退避重试、超时降级等容错策略，确保系统在 API 不可用时仍能维持基本功能。

## 五、预期结果及其意义

### 5.1 预期结果

1. 完成 MOSAIC 系统的完整架构设计与核心模块实现，包括三层架构、抽象接口层、Agent 核心调度管道、LLM Provider、能力插件、ROS 2 通信适配层
2. 实现基于 LLM Function Calling 的自然语言任务解析，支持单任务映射和多步骤任务分解
3. 实现 ARIA 多层记忆架构和 EmbodiedRAG 检索机制的原型验证
4. 实现导航、建图、运动等能力插件，并在 Gazebo 仿真环境中完成 TurtleBot3 的功能验证
5. 验证系统的核心正确性属性：Capability 注册-解析 round-trip、任务状态机合法转换、管道错误传播等
6. 完成毕业设计论文一篇

### 5.2 研究意义

**工程实践意义**：填补了 ROS 2 生态中缺少基于 LLM 的通用任务调度中间层的空白。系统通过 Function Calling 机制将 LLM 的任务理解能力与 ROS 2 的 Action 执行能力桥接，实现了机器人任务调度从"开发者手写行为树"到"用户说自然语言"的范式转换，显著降低了机器人任务配置的复杂度和使用门槛。

**学术探索意义**：将具身智能领域的前沿研究成果（SayCan 的能力锚定思想、EmbodiedRAG 的场景图检索、SayPlan 的 3D 场景图表征）进行了面向 ROS 2 生态的工程化落地与融合创新，验证了 LLM 作为机器人任务调度"大脑"的可行性，为后续研究（VLA 集成、多机器人协作）提供了可扩展的系统基座。

**社会应用意义**：系统降低了机器人操控的技术门槛，使非专业人员能够通过自然语言与机器人交互，对智慧养老、物业管理、安防巡检等服务机器人应用场景具有直接的推广价值。
