# 产品需求设计文档

## 面向 ROS 2 架构的 AI Agent 机器人任务调度系统

---

## 1 产品定位

一个面向 ROS 2 移动机器人的智能任务调度中间层系统。用户通过自然语言下达指令，系统自动完成任务理解、分解、规划，并通过 ROS 2 原生通信接口（Topic / Service / Action）驱动机器人执行，无需修改底层控制逻辑。

**核心价值**：
- 用自然语言替代行为树/状态机的手动配置
- AI Agent 与 ROS 2 通信机制原生适配，开箱即用
- 降低机器人任务调度的使用门槛
- 插件化架构，支持从传统导航到端到端视觉导航的平滑演进

---

## 2 用户角色

| 角色 | 描述 | 典型场景 |
|------|------|---------|
| 操作员 | 非专业开发人员，通过自然语言与机器人交互 | "去厨房拿一杯水" |
| 开发者 | ROS 2 开发人员，注册新的机器人能力和扩展系统 | 注册新的 Action Server、定义工具描述 |
| 系统管理员 | 负责系统部署、API 配置和运维 | 配置 LLM API 密钥、监控系统状态 |

## 3 系统架构总览

### 3.1 架构设计原则

1. **感知-决策-执行分离**：感知输入、智能决策（LLM）、机器人执行（ROS 2 Action）三层解耦，任意一层可独立替换
2. **Provider 插件机制**：AI 模型调用通过统一的 Provider 接口抽象，当前实现 LLMProvider，未来可插入 VLM/端到端导航 Provider
3. **ROS 2 原生适配**：通过 Topic / Service / Action 三种适配器与 ROS 2 生态无缝对接
4. **能力插件化（Capability Plugin）**：导航、建图、旋转等机器人能力均为可插拔的 Capability 插件，Agent 核心仅依赖抽象的 Capability 接口，不感知具体实现。导航只是众多能力之一，未来可整体替换导航方案而不影响系统其他部分
5. **ROS 2 版本兼容性设计**：系统基于 ROS 2 Humble 开发验证，但架构设计面向跨版本兼容，确保可平滑迁移至 Jazzy 及后续发行版
6. **依赖倒置**：Agent 核心层定义接口（抽象），具体实现（Provider、Capability、Adapter）依赖接口而非反向依赖。任何模块的替换只需实现对应接口，无需修改调用方代码


### 3.2 ROS 2 版本兼容性策略

当前开发基于 ROS 2 Humble（LTS, 支持至 2027），同时为 Jazzy（LTS, 支持至 2029）及后续版本预留兼容路径：

**兼容性设计要点**：

- 仅使用 ROS 2 稳定公共 API（`rclpy` 核心接口），不依赖发行版特有的内部实现
- ROS 2 通信适配层封装所有与 ROS 2 版本相关的调用（Topic / Service / Action），上层 Agent 核心不直接调用 `rclpy`，版本差异在适配层内部吸收
- 仿真环境通过 launch 文件隔离：Humble 使用 Gazebo Classic，Jazzy 使用 Gazebo Harmonic，Agent 核心代码无需修改
- 配置文件中声明 ROS 2 版本相关参数（如 QoS 策略、DDS 配置），避免硬编码
- Nav2 / SLAM Toolbox 等依赖包通过 `package.xml` 声明版本范围，而非锁定特定版本

**Humble → Jazzy 已知差异与应对**：

| 差异项 | Humble | Jazzy | 应对策略 |
|--------|--------|-------|---------|
| 仿真器 | Gazebo Classic | Gazebo Harmonic (gz-sim) | launch 文件分版本维护，Agent 不感知仿真器差异 |
| DDS 中间件 | Fast-DDS (默认) | Fast-DDS (默认，可选 Zenoh) | 通过 RMW 环境变量切换，代码无需修改 |
| QoS 默认策略 | 部分话题默认 BEST_EFFORT | 更严格的 QoS 匹配 | 适配层显式声明 QoS Profile，不依赖默认值 |
| 消息序列化 | CDR | CDR（Iron+ 引入 Type Hash） | 使用标准消息类型，不自定义序列化 |
| Python API | rclpy (稳定) | rclpy (兼容，少量新增) | 仅使用两版本共有的稳定 API 子集 |

**目录结构体现**：

```
launch/
├── humble/                    # Humble 专用 launch 配置
│   └── gazebo_classic.launch.py
├── jazzy/                     # Jazzy 专用 launch 配置（预留）
│   └── gazebo_harmonic.launch.py
├── agent_launch.py            # Agent 核心启动（版本无关）
└── navigation_launch.py       # 导航启动（版本无关）
```

### 3.4 核心抽象接口定义

系统通过三层抽象接口实现完全解耦，任何模块的替换只需实现对应接口：

#### 3.4.1 ModelProvider — AI 模型抽象

```python
class ModelProvider(ABC):
    """AI 模型提供者抽象接口
    职责边界：仅负责"理解意图"，不涉及执行层逻辑
    """

    @abstractmethod
    async def parse_task(self, context: TaskContext) -> TaskResult:
        """解析自然语言指令，返回结构化任务描述
        输入：用户文本（未来可扩展图像/语音）
        输出：意图 + 参数的结构化描述
        """
        pass

    @abstractmethod
    def get_supported_intents(self) -> list[str]:
        """声明该 Provider 支持的意图类型列表
        用于 Agent 核心校验任务是否可处理
        """
        pass


class LLMProvider(ModelProvider):
    """V1 当前实现：基于纯文本 LLM 的任务解析
    文本指令 → OpenAI GPT Function Calling → 结构化任务
    """
    ...
```

#### 3.4.2 Capability — 机器人能力抽象

```python
class Capability(ABC):
    """机器人能力抽象接口
    每种能力（导航、建图、旋转等）实现此接口
    Agent 核心仅通过此接口调度，不感知具体实现
    """

    @abstractmethod
    def get_name(self) -> str:
        """能力唯一标识名（如 'navigation', 'mapping'）"""
        pass

    @abstractmethod
    def get_supported_intents(self) -> list[str]:
        """该能力可处理的意图列表（如 ['navigate_to', 'patrol']）"""
        pass

    @abstractmethod
    async def execute(self, task: Task) -> ExecutionResult:
        """执行任务，返回执行结果
        内部自行决定如何调用 ROS 2 适配层
        """
        pass

    @abstractmethod
    async def cancel(self) -> bool:
        """取消当前正在执行的任务"""
        pass

    @abstractmethod
    async def get_status(self) -> CapabilityStatus:
        """返回当前能力的状态（空闲/执行中/错误等）"""
        pass


class NavigationCapability(Capability):
    """V1 实现：基于 Nav2 的导航能力
    未来可整体替换为端到端视觉导航实现
    内部封装：Nav2 Action 调用、地图管理、定位监控、语义地名映射
    """
    ...

class MappingCapability(Capability):
    """V1 实现：基于 SLAM Toolbox 的建图能力
    未来可替换为其他 SLAM 方案或视觉建图方案
    """
    ...

class MotionCapability(Capability):
    """基础运动能力：旋转、平移等"""
    ...
```

#### 3.4.3 CapabilityRegistry — 能力注册与发现

```python
class CapabilityRegistry:
    """能力注册中心
    - 管理所有已注册的 Capability 插件
    - 支持运行时动态注册/注销
    - 根据意图类型查找对应的 Capability
    """

    def register(self, capability: Capability) -> None:
        """注册一个能力插件"""
        ...

    def unregister(self, name: str) -> None:
        """注销一个能力插件"""
        ...

    def resolve(self, intent: str) -> Capability:
        """根据意图类型查找能处理该意图的 Capability
        如 'navigate_to' → NavigationCapability
        """
        ...

    def list_capabilities(self) -> list[CapabilityInfo]:
        """列出所有已注册能力及其支持的意图"""
        ...
```

**模块间调用关系（依赖方向）**：

```
Agent 核心 ──依赖──→ ModelProvider 接口 ←──实现── LLMProvider
Agent 核心 ──依赖──→ Capability 接口   ←──实现── NavigationCapability
Agent 核心 ──依赖──→ CapabilityRegistry
Capability ──依赖──→ ROS2 Adapter 接口 ←──实现── Topic/Service/Action Adapter
```

Agent 核心永远只依赖左侧的抽象接口，右侧的具体实现可自由替换。


### 3.5 演进路线图

| 阶段 | 导航范式 | 地图依赖 | Provider | Navigation Capability | 参考技术 |
|------|---------|---------|----------|----------------------|---------|
| V1（当前） | LLM → 坐标 → Nav2 | SLAM 几何地图 | LLMProvider | Nav2NavigationCapability | Nav2 + SLAM Toolbox |
| V2（近期） | VLM + 语义记忆 → Nav2 | SLAM 地图 + 语义层 | VLMProvider | Nav2NavigationCapability + SemanticMemory | GPT-4V / LLaVA |
| V3（远期） | 视频+指令 → 端到端动作 | 无地图（仅单目 RGB） | E2ENavProvider | E2ENavigationCapability（替换整个导航能力） | NaVid (RSS 2024) / RT-2 |

V3 阶段参考 NaVid（RSS 2024）等端到端 VLN 方案：仅依赖单目 RGB 视频流 + 自然语言指令即可实现导航，无需 SLAM 地图、里程计或深度传感器。届时只需实现新的 `E2ENavigationCapability` 替换现有导航能力，Agent 核心代码零修改。SLAM 可降级为可选的安全兜底 Capability。

## 4 功能需求

### 4.1 自然语言任务输入（P0 - 必须实现）

**FR-001 自然语言指令接收**
- 系统应提供 CLI 命令行界面接收用户自然语言输入
- 支持中文和英文指令
- 输入示例："导航到A点"、"先去A点，再去B点"、"巡逻A、B、C三个点"

**FR-002 多轮对话支持**
- 当指令不明确时，系统应主动向用户确认
- 支持上下文关联的连续指令（如"再去一次"）

### 4.2 任务解析模块（P0 - 必须实现）

**FR-003 意图识别**
- 系统应能识别以下核心意图类型：
  - `navigate_to` — 导航到指定位置
  - `patrol` — 巡逻（多点循环导航）
  - `rotate` — 原地旋转指定角度
  - `stop` — 停止当前任务
  - `status` — 查询机器人当前状态
  - `start_mapping` — 启动 SLAM 建图
  - `save_map` — 保存当前地图
  - `stop_mapping` — 停止建图
- 意图列表由已注册的 Capability 动态提供（通过 `CapabilityRegistry.list_capabilities()`），而非硬编码在 Agent 核心中
- 通过调用 ModelProvider（当前为 LLMProvider）实现意图识别

**FR-004 参数提取**
- 从自然语言中提取任务参数：
  - 目标位置（支持语义地名 → 坐标映射，如"厨房" → [2.0, 3.5, 0.0]）
  - 运动参数（速度、角度等）
  - 任务约束（顺序、重复次数等）

**FR-005 任务结构化输出**
- 将解析结果转化为标准化的任务描述格式（JSON）：
```json
{
  "task_id": "uuid",
  "intent": "navigate_to",
  "params": {
    "target": {"x": 2.0, "y": 3.5, "theta": 0.0},
    "target_name": "厨房"
  },
  "priority": 1,
  "timestamp": "2026-02-12T10:00:00Z"
}
```


### 4.3 任务规划模块（P0 - 必须实现）

**FR-006 单任务规划**
- 将单一意图通过 `CapabilityRegistry.resolve(intent)` 查找对应 Capability，映射为执行调用

**FR-007 多步骤任务分解**
- 将复合指令分解为有序的子任务序列
- 示例："先去A点，然后去B点" → [navigate_to(A), navigate_to(B)]
- 每个子任务独立通过 Registry 解析到对应 Capability

**FR-008 任务队列管理**
- 维护待执行任务的优先级队列
- 支持任务的插入、取消和重排序

### 4.4 任务执行模块（P0 - 必须实现）

**FR-009 Capability 调度**
- 任务执行器通过 `Capability.execute(task)` 统一调度，不直接操作 ROS 2 Action Client
- Capability 内部自行决定如何与 ROS 2 适配层交互（可能调用 Action、Service 或 Topic）
- 支持的执行操作：
  - 发送任务（execute）
  - 接收反馈（通过 Capability 回调）
  - 获取结果
  - 取消任务（cancel）

**FR-010 任务状态监控**
- 实时跟踪任务执行状态：PENDING → EXECUTING → SUCCEEDED / FAILED / CANCELLED
- 任务失败时支持自动重试（可配置重试次数）
- 状态通过 `Capability.get_status()` 统一查询

**FR-011 执行反馈上报**
- Capability 通过统一的回调接口上报执行反馈
- 反馈信息转化为用户可读的文本输出

### 4.5 外部 AI 模型 API 调用模块（P0 - 必须实现）

**FR-012 LLM API 通信**
- 通过 HTTP/HTTPS 协议调用 OpenAI GPT API
- 使用 Function Calling 机制定义机器人可用能力
- Function 定义由 `CapabilityRegistry` 动态生成（已注册的 Capability 自动暴露为 LLM 可调用的 Function）
- 支持异步调用，不阻塞 ROS 2 节点运行

**FR-013 API 配置管理**
- 支持通过配置文件（YAML）管理 API 参数：
  - API endpoint / API key
  - 模型选择（gpt-4 / gpt-3.5-turbo 等）
  - 超时时间 / 重试策略
  - Token 限制

**FR-014 错误处理与降级**
- API 调用失败时的处理策略：
  - 自动重试（指数退避）
  - 超时降级（使用本地规则匹配作为备选）
  - 错误日志记录

### 4.6 ROS 2 通信适配层（P0 - 必须实现）

本层是实现 ROS 2 版本兼容的关键隔离层。Capability 插件通过适配层与 ROS 2 交互，Agent 核心不直接调用 `rclpy`，版本差异在此层内部吸收。

**FR-015 Topic 适配器**
- 订阅机器人状态话题（如 `/odom`、`/scan`、`/amcl_pose`）获取环境信息
- 发布系统状态话题供外部监控

**FR-016 Service 适配器**
- 调用 ROS 2 Service 完成即时查询操作（如获取地图信息、查询参数）

**FR-017 Action 适配器**
- 封装统一的 Action Client 接口
- 支持动态注册新的 Action 类型
- 管理 Action 的生命周期（发送目标 → 监听反馈 → 获取结果）
- Capability 插件通过 Action 适配器与 ROS 2 Action Server 交互，而非直接创建 Action Client


### 4.7 导航能力插件（P0 - 必须实现，作为 Capability 实现）

导航不再是系统的特殊模块，而是 `NavigationCapability` 插件的内部实现细节。以下功能均封装在该 Capability 内部，Agent 核心不感知。

**FR-018 SLAM 建图（MappingCapability 内部）**
- 集成 SLAM Toolbox 作为默认 2D SLAM 引擎
- 支持通过自然语言触发建图操作："开始建图"、"保存地图"、"停止建图"

**FR-019 地图管理（NavigationCapability 内部）**
- 管理地图文件的加载、保存和切换（`.pgm` + `.yaml` 格式）
- 支持多地图管理（如不同楼层），通过自然语言切换："加载办公室地图"

**FR-020 定位状态监控（NavigationCapability 内部）**
- 订阅 `/amcl_pose` 获取实时定位，监控定位质量
- 定位丢失时暂停任务并尝试重定位，恢复后自动继续

**FR-021 语义地名映射（独立服务，供多个 Capability 共享）**
- 维护语义地名到地图坐标的映射表（`locations.yaml`）：
```yaml
locations:
  厨房: {x: 2.0, y: 3.5, theta: 0.0}
  客厅: {x: 0.0, y: 0.0, theta: 0.0}
  卧室: {x: -1.0, y: 2.0, theta: 1.57}
  充电桩: {x: 0.0, y: 0.0, theta: 0.0}
```
- 支持通过自然语言动态标注："把这里标记为会议室" → 读取当前定位坐标并保存
- 支持运行时动态添加/修改/删除地点
- 语义地名服务作为共享服务注入到需要的 Capability 中，而非绑定在某个特定 Capability 内部

### 4.8 系统监控与日志（P1 - 应该实现）

## 5 非功能需求

### 5.1 性能需求

| 指标 | 要求 |
|------|------|
| 自然语言解析延迟 | ≤ 3 秒（含 API 调用） |
| 任务调度响应时间 | ≤ 500 毫秒（从解析完成到 Capability.execute 调用） |
| 系统启动时间 | ≤ 10 秒（含所有 Capability 注册） |
| 并发任务队列容量 | ≥ 20 个待执行任务 |

### 5.2 可靠性需求

- API 调用失败自动重试，最多 3 次
- 网络断开时系统不崩溃，进入降级模式
- 任务执行失败后自动恢复到安全状态
- 单个 Capability 异常不影响其他 Capability 和 Agent 核心运行

### 5.3 可扩展性需求

- 新增机器人能力通过实现 `Capability` 接口 + 注册到 `CapabilityRegistry`，无需修改 Agent 核心代码
- 新 Capability 注册后自动暴露为 LLM 可调用的 Function
- 支持替换不同的 LLM 提供商（OpenAI / 本地模型），通过实现 `ModelProvider` 接口
- 导航方案可整体替换（Nav2 → 端到端视觉导航），只需实现新的 `NavigationCapability`
- 模块间通过抽象接口松耦合，依赖方向始终从具体实现指向抽象接口
- Agent 核心代码与 ROS 2 发行版解耦，通过适配层隔离版本差异

### 5.4 安全性需求

- API 密钥不硬编码，通过环境变量或加密配置文件管理
- 对用户输入进行基本的安全过滤（防止 prompt injection）

| 变更点 | 旧架构 | 新架构 |
|--------|--------|--------|

| Function Calling | 手动维护 | 从 CapabilityRegistry 动态生成 |

## 8 核心工作流

### 8.1 标准任务执行流程

```
用户输入 "导航到厨房"
       │
       ▼
[交互接口] 接收文本
       │
       ▼
[任务解析] 调用 ModelProvider.parse_task()
       │  → 识别意图: navigate_to
       │  → 提取参数: target_name="厨房"
       │
       ▼
[任务规划] CapabilityRegistry.resolve("navigate_to")
       │  → 找到 NavigationCapability
       │  → 语义映射: "厨房" → {x: 2.0, y: 3.5, theta: 0.0}
       │
       ▼
[任务队列] 入队，等待调度
       │
       ▼
[任务执行] NavigationCapability.execute(task)
       │  → 内部调用 Action Adapter → Nav2 NavigateToPose
       │  → 监听 Feedback → 上报: "正在前往厨房，剩余距离 2.3m"
       │
       ▼
[执行完成] 返回 ExecutionResult: "已到达厨房"
```

### 8.2 多步骤任务流程

```
用户输入 "先去厨房，再去客厅，最后回到充电桩"
       │
       ▼
[任务解析] ModelProvider 分解为 3 个子任务
       │
       ▼
[任务规划] 每个子任务通过 Registry 解析:
       │  1. resolve("navigate_to") → NavigationCapability
       │  2. resolve("navigate_to") → NavigationCapability
       │  3. resolve("navigate_to") → NavigationCapability
       │
       ▼
[任务队列] 按序入队
       │
       ▼
[任务执行] 依次调用 Capability.execute()，每个完成后自动触发下一个
```

### 8.3 Capability 替换流程（开发者视角）

```
场景：将 Nav2 导航替换为端到端视觉导航

1. 实现新的 Capability:
   class E2ENavigationCapability(Capability):
       def get_name(self) -> str: return "navigation"
       def get_supported_intents(self) -> list[str]: return ["navigate_to", "patrol"]
       async def execute(self, task: Task) -> ExecutionResult: ...  # 端到端导航逻辑

2. 修改配置文件 agent_config.yaml:
   capabilities:
     navigation:
       implementation: "capabilities.navigation.e2e_capability.E2ENavigationCapability"
       # 原: "capabilities.navigation.nav2_capability.Nav2NavigationCapability"

3. 重启系统 → Agent 核心代码零修改，自动使用新导航方案
```
