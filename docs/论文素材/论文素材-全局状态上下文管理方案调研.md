# 论文素材：机器人全局状态上下文管理方案调研

## 1. 问题定义

LLM-centric 机器人任务规划系统面临一个核心矛盾：环境表征的丰富性与 LLM 上下文窗口的有限性之间的冲突。

随着机器人操作环境从单房间扩展到多楼层建筑，3D 场景图中的实体数量可能从数十增长到数千。直接将完整场景图序列化为 LLM 输入会导致：
- token 数量超出模型上下文窗口限制（如 GPT-4o 的 128k token）
- LLM 注意力偏差：研究表明 LLM 对输入中间位置的信息关注度显著低于首尾位置（"Lost in the Middle" 现象，Liu et al., 2024）
- 任务无关信息干扰规划质量（Shi et al., ICML 2023 证实 LLM 易被无关上下文分散注意力）
- 规划延迟随 token 数量线性增长

本调研聚焦于：如何设计一个前沿的、泛化性强的全局状态存储与检索架构，使 LLM 规划器始终获得精简、相关、充分的环境上下文。

## 2. 前沿方案综述

### 2.1 SayPlan：层级折叠/展开（CoRL 2023）

SayPlan（Rana et al., CoRL 2023）首次将 3D 场景图引入 LLM 规划，提出语义搜索机制：LLM 通过 `expand(node)` / `contract(node)` API 在层级化场景图上手动展开/折叠子图，逐步定位任务相关区域。

局限性：
- 压缩效果依赖场景图的层级结构，底层（如房间内所有物体）压缩有限
- LLM 需要理解图结构并执行多轮展开/折叠操作，增加规划步骤和 token 消耗
- 假设场景图已预先完整构建，不支持在线增量构建
- 无反馈机制：检索过程是开环的，无法根据规划中间结果动态调整

来源：[SayPlan: Grounding Large Language Models Using 3D Scene Graphs for Scalable Robot Task Planning](https://arxiv.org/abs/2306.07461)

### 2.2 EmbodiedRAG：任务驱动的 3D 场景图子图检索（2024）

EmbodiedRAG（Booker et al., 2024）将 RAG 范式适配到具身智能领域，提出 3D 场景图子图检索框架。核心创新是将场景图节点视为"文档"进行向量化索引，通过任务驱动的语义检索自动提取相关子图，替代 SayPlan 的手动折叠/展开。

关键技术组件：

（1）场景图文档索引（Document Indexing）：将场景图中每个实体节点视为一个"文档"，节点的语义标签作为文档内容，属性作为元数据，嵌入到向量存储中（如 ChromaDB、FAISS）。场景图天然具有"分块"结构（每个节点即一个 chunk），无需额外的文档分割策略。

（2）LLM 引导的预检索抽象（Pre-Retrieval）：在规划开始前，LLM 仅基于任务文本（无需环境知识）推断可能相关的实体类型和属性。例如"煮鸡蛋"任务，LLM 预抽象出 {鸡蛋, 锅, 加热源} 及相关属性 {温度, 可烹饪性}。这些预抽象实体作为向量检索的初始查询。

（3）子图 Grounding：以向量检索命中的节点为锚点，提取诱导子图（包含节点间所有关系边），并过滤仅保留任务相关属性。这确保 LLM 不仅获得相关实体，还获得它们之间的空间和语义关系。

（4）Self-Query 反馈机制：LLM 规划过程中产生的"思考"（thoughts）被解析为新的检索查询，动态扩展检索范围。例如 LLM 在规划"煮鸡蛋"时意识到需要"翻转工具"，Self-Query 机制自动检索"铲子"、"锅铲"等实体加入子图。这是 EmbodiedRAG 相对于传统 RAG 的关键创新——闭环检索。

实验结果（AI2Thor 模拟环境，GPT-4o-mini）：
- 累计 token 消耗降低 90%
- 平均每步规划时间减少 70%
- 任务成功率提升（尤其在大规模环境中）
- 在 1135 个干扰物体的环境中，EmbodiedRAG-strict 的性能不随干扰物体数量增加而下降

硬件验证：在 Boston Dynamics Spot 四足机器人 + NVIDIA Jetson AGX Orin 上部署，使用 ConceptGraphs 构建 3DSG，llama3.1:8b 本地推理。Full-Mem 方案在大环境中因 token 过载导致严重幻觉，而 EmbodiedRAG-strict 未出现幻觉。

来源：[EmbodiedRAG: Dynamic 3D Scene Graph Retrieval for Efficient and Scalable Robot Task Planning](https://arxiv.org/abs/2410.23968)

### 2.3 ReMEmbR：长时时空语义记忆（NVIDIA, 2024）

ReMEmbR（NVIDIA, 2024）解决的是另一个维度的问题：机器人在长时间部署（数小时到数天）中如何构建和查询语义记忆。

核心架构分为两个阶段：

（1）记忆构建阶段：机器人运行过程中，短视频片段被 VLM（NVIDIA VILA）生成语义描述（caption），连同时间戳和坐标信息一起嵌入到 MilvusDB 向量数据库中。这种方式使记忆能够捕获 VLM 可描述的任何信息——从动态事件（人走动）到具体小物体。

（2）查询阶段：LLM Agent 作为推理引擎，通过多种工具函数迭代查询向量数据库——文本查询、时间查询、位置查询——直到回答用户问题。这种 Agent 循环查询模式使机器人能够进行时空推理（"最近在哪里看到过钥匙？"）。

关键设计决策：
- 使用向量数据库而非直接存储视频，解决长上下文推理的效率问题
- VLM 在设备端运行（Jetson Orin），不依赖网络连接
- 记忆格式可扩展——向量数据库可以轻松添加新类型的信息

在 Nova Carter 机器人上的部署验证了该方案在真实环境中的可行性，机器人能够回答"带我去最近的电梯"、"带我去能买零食的地方"等需要时空推理的查询。

来源：[ReMEmbR: Building and Reasoning Over Long-Horizon Spatio-Temporal Memory for Robot Navigation](https://arxiv.org/abs/2409.13682)

### 2.4 GraphRAG：知识图谱增强的检索生成（Microsoft, 2024）

GraphRAG（Edge et al., Microsoft Research, 2024）虽然面向文本领域，但其核心思想对机器人场景图检索有重要启发。

GraphRAG 将传统 RAG 的"文档块检索"升级为"知识图谱 + 社区摘要检索"：
- 从源文档中提取实体和关系构建知识图谱
- 对图进行社区检测（Leiden 算法），生成层级化社区结构
- 每个社区生成 LLM 摘要
- Local Search：从实体出发，检索相关社区和邻居
- Global Search：跨社区聚合摘要，回答需要全局理解的问题

对机器人场景图的启发：场景图天然具有社区结构（房间即社区），可以为每个房间生成语义摘要，支持"厨房里有什么？"这类全局查询。

来源：[From Local to Global: A Graph RAG Approach to Query-Focused Summarization](https://arxiv.org/abs/2404.16130)

### 2.5 ConceptGraphs：开放词汇 3D 场景图（ICRA 2024）

ConceptGraphs（Gu et al., ICRA 2024）提出了开放词汇 3D 场景图的构建方法，是 EmbodiedRAG 等上层检索框架的底层表征基础。

核心方法：利用 2D 基础模型（如 SAM、CLIP）的输出，通过多视角关联融合到 3D，构建开放词汇的图结构化场景表征。该表征可泛化到训练时未见过的语义类别，无需收集大规模 3D 数据集或微调模型。

来源：[ConceptGraphs: Open-Vocabulary 3D Scene Graphs for Perception and Planning](https://arxiv.org/abs/2309.16650)

### 2.6 Clio：实时任务驱动的开放集 3D 场景图（MIT, 2024）

Clio（Maggio et al., MIT SPARK Lab, 2024）从另一个角度解决信息过载问题：在场景图构建阶段就进行任务驱动的压缩，而非构建完整场景图后再检索。

核心思想：给定自然语言任务列表，Clio 在构建 3D 场景图时使用信息瓶颈方法，自动选择与任务相关的语义粒度和物体子集。例如，对于"拿起桌上的杯子"任务，Clio 会保留杯子和桌子的详细信息，而将其他物体聚合为粗粒度表示。

与 EmbodiedRAG 的互补性：Clio 在感知端压缩（构建时过滤），EmbodiedRAG 在检索端压缩（查询时过滤），两者可以组合使用。

来源：[Clio: Real-time Task-Driven Open-Set 3D Scene Graphs](https://arxiv.org/abs/2404.13696)

### 2.7 DovSG：动态开放词汇场景图（2024）

DovSG（2024）专注于长期移动操作场景中场景图的动态更新问题。使用 VLM 进行物体检测获取高层语义特征，基于分割物体生成结构化 3D 场景图表示低层空间关系，并结合语言引导的任务规划模块实现长期任务执行。

关键贡献：解决了场景图在长期部署中的动态一致性维护问题——物体被移动、添加或移除后，场景图能够正确更新。

来源：[Dynamic Open-Vocabulary 3D Scene Graphs for Long-term Language-Guided Mobile Manipulation](https://arxiv.org/abs/2410.11989)

## 3. 方案对比分析

| 维度 | SayPlan | EmbodiedRAG | ReMEmbR | GraphRAG | Clio |
|---|---|---|---|---|---|
| 检索方式 | LLM 手动展开/折叠 | 向量相似度 + 子图 Grounding | 向量数据库多模态查询 | 知识图谱 + 社区摘要 | 构建时任务驱动压缩 |
| 压缩阶段 | 查询时 | 查询时 | 查询时 | 索引时 + 查询时 | 构建时 |
| 场景图要求 | 预构建完整层级图 | 支持在线增量构建 | 无场景图（视频记忆） | 从文本构建知识图谱 | 实时构建压缩图 |
| 反馈机制 | 无 | Self-Query 闭环 | Agent 迭代查询 | 无 | 无 |
| 时间维度 | 无 | 动态适应环境变化 | 长时时空推理 | 无 | 实时 |
| token 压缩率 | 中等（依赖层级） | 90%（论文数据） | 高（仅检索相关片段） | 高 | 高（构建时已压缩） |
| 经验利用 | 无 | 无 | 有（历史记忆） | 有（社区摘要） | 无 |
| 硬件验证 | 无 | Spot + Jetson | Nova Carter | 无（文本领域） | 真实环境验证 |

## 4. MOSAIC 系统中的 ARIA 模块：融合方案

基于上述调研，MOSAIC 系统的全局状态管理模块 ARIA（Agent with Retrieval-augmented Intelligence Architecture）设计了三层记忆架构，融合 EmbodiedRAG 的任务驱动子图检索、ReMEmbR 的长时语义记忆、GraphRAG 的知识图谱混合检索思想：

### 4.1 三层记忆模型

（1）工作记忆（Working Memory）：类比人类短期记忆，存储机器人实时状态（位姿、传感器数据、能力状态），内存数据结构，实时覆写。对应 RobotState。

（2）语义记忆（Semantic Memory）：类比人类长期知识，存储 3D 场景图节点/边及其向量嵌入索引。场景图节点在写入时自动向量化（参考 EmbodiedRAG 的 Document Indexing），支持语义相似度检索。图结构 + VectorStore 双存储，持久化且增量更新。

（3）情景记忆（Episodic Memory）：类比人类经验回忆，存储任务执行历史（成功/失败经验、重规划记录、环境快照）。向量化存储支持相似任务经验召回（参考 ReMEmbR 的长时记忆思想）。按时间衰减，近期经验权重更高。

### 4.2 EmbodiedRAG 检索流程

TaskPlanner 不直接读取全量 SceneGraph，而是通过 `retrieve_context()` 接口获取任务驱动的精简上下文：

1. Pre-Retrieval：LLM 从任务文本推断相关实体和属性（无需环境知识）
2. 向量检索：在 VectorStore 中检索与预抽象实体最相似的场景图节点
3. 子图 Grounding：以检索节点为锚点提取诱导子图 + 过滤任务相关属性
4. Self-Query：规划过程中 LLM 的 thoughts 反馈给检索器，动态扩展检索范围
5. 经验召回：从情景记忆检索相似任务的历史经验
6. 组装 PlanningContext 返回给 TaskPlanner

### 4.3 方案优势

- 泛化性：向量检索天然支持开放词汇，无需预定义实体类别
- 可扩展性：环境规模增长时，检索复杂度为 O(log n)，不随场景图大小线性增长
- 闭环检索：Self-Query 机制使检索过程能够根据规划中间结果动态调整
- 经验积累：情景记忆使系统能够从历史执行中学习，避免重复犯错
- 多模态兼容：向量存储可以索引文本嵌入、图像嵌入（CLIP）、场景图节点嵌入等多种模态
